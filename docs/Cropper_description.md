# Редактор обрезки и логика координат

## Архитектура хранения данных

В системе используются **два хранилища** для каждого изображения:

### 1. Файловое хранилище

| Папка | Назначение |
|-------|------------|
| `data/originals/` | Неизменяемые оригиналы. Всегда открываются в редакторе обрезки как подложка. |
| `data/images/` | Результат обрезки. Именно эти файлы видит нейросеть (YOLO) и редакторы сегментов/текста. |

### 2. База данных (SQLite)

| Таблица | Описание |
|---------|----------|
| `images` | Метаданные: пути к файлам, статус (`crop`, `cropped`, `segment`, `texted`), `crop_params` (JSON) |
| `annotations` | Аннотации: полигоны и распознанный текст в JSON-поле `polygons` |

**«Источник истины»:** SQLite база данных (`database.db`), а не файлы.

---

## 1. Пространства координат

1. **Экран (Viewport)**: Координаты в браузере (зависят от зума и положения камеры).
2. **Мир (World/Image)**: Координаты в пикселях **оригинального** изображения. При сохранении кропа из JS на сервер всегда уходят именно эти координаты.
3. **Локальные (Local)**: Координаты сегментов (строк текста) внутри уже обрезанного изображения.

---

## 2. Логика пересчета при повторной обрезке

Это самая важная часть системы. Если изображение уже было обрезано и размечено, а пользователь решил **изменить обрезку еще раз**, система выполняет трехэтапный пересчет координат для всех существующих сегментов:

### Этап 1: Локальные → Глобальные
На основе старых `crop_params` точки переводятся из координат «старого маленького холста» в абсолютные координаты оригинального изображения через билинейную интерполяцию `lerp_quad()`.

### Этап 2: Трансформация
Исходное изображение (`original`) обрезается по новым углам с помощью Pillow `Image.QUAD`.

### Этап 3: Глобальные → Новые локальные
Точки оригинала проецируются на «новый маленький холст» через `get_uv()`.

**Результат:** «Движение» рамки обрезки так, чтобы уже нарисованные полигоны строк текста «прилипали» к бумаге и не уплывали при поворотах.

## 3. Сохранность данных (Smart Save)

Все операции в системе работают по принципу **Update, а не Replace**:

| Операция | Что обновляется | Что сохраняется |
|----------|-----------------|-----------------|
| **Сохранение сегментов** | `annotations.polygons` | `images.crop_params`, `images.status` |
| **Обрезка** | `images.crop_params`, `images.status` | `annotations.polygons` (пересчитываются) |
| **Распознавание текста** | `annotations.polygons[].text` | `annotations.polygons[].points` |

### Механизм

1. Бэкенд загружает текущие записи из БД (`images` + `annotations`)
2. Обновляются только пришедшие поля
3. Поля, не участвующие в текущей операции, **бережно сохраняются**

### Реализация

`AnnotationService.save_annotation()` в `services/annotation_service.py`:

```python

    # Обновляет image.crop_params и annotation.polygons
    # Сохраняет существующие данные
```

## 4. Параметры Pillow QUAD

Для корректной работы трансформации используется метод `Image.QUAD`.

### Порядок передачи углов в Pillow

| Индекс | Угол | Описание |
|--------|------|----------|
| `0` | Top-Left | Верхний левый |
| `1` | Bottom-Left | Нижний левый |
| `2` | Bottom-Right | Нижний правый |
| `3` | Top-Right | Верхний правый |

> **Примечание:** Фронтенд на Fabric.js отправляет углы в порядке обхода по часовой стрелке (TL, TR, BR, BL), поэтому на бэкенде в `perform_crop()` индексы 1 и 3 меняются местами для соответствия спецификации Pillow.

## 5. Структура данных в БД

### Таблица `images`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer | Первичный ключ |
| `project_id` | Integer | Внешний ключ на проект |
| `filename` | String | Имя файла |
| `original_path` | String | Путь к оригиналу |
| `cropped_path` | String | Путь к обрезанному файлу |
| `status` | String | `crop`, `cropped`, `segment`, `texted` |
| `crop_params` | JSON | `{x, y, width, height, angle, corners}` |

### Таблица `annotations`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer | Первичный ключ |
| `image_id` | Integer | Внешний ключ на изображение |
| `polygons` | JSON | Массив полигонов: `[{points: [[x1,y1],...], text: ''}]` |

### Пример JSON в `annotations.polygons`

```json
[
  {
    "points": [[100, 100], [200, 100], [200, 150], [100, 150]],
    "text": "Распознанный текст"
  },
  {
    "points": [[50, 200], [300, 200], [300, 250], [50, 250]],
    "text": "Другая строка"
  }
]
```

## 6. Компоненты системы

| Компонент | Файл | Ответственность |
|-----------|------|-----------------|
| **AnnotationService** | `services/annotation_service.py` | Бизнес-логика работы с аннотациями |
| **AnnotationRepository** | `database/repository/annotation_repository.py` | Доступ к таблице `annotations` |
| **ImageRepository** | `database/repository/image_repository.py` | Доступ к таблице `images` |
| **logic.py** | `logic.py` | Низкоуровневая логика: координаты, кроп, XML |

---

## 7. Обратная совместимость

Папка `data/annotations/` **не используется** в новой версии. Аннотации хранятся только в SQLite.

При первом запуске `init_db.py` может выполнить миграцию существующих JSON-аннотаций в базу данных (если папка существует).
